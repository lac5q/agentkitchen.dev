<objective>
Create the persistence and provider foundation for LLM semantic recall: the
`message_embeddings` SQLite table, the Ollama `nomic-embed-text` embedding
provider (with a typed degraded fallback), and a thin store module for reading
and writing embedding rows.

Purpose: RECALL-01 (semantic/hybrid modes) and RECALL-02 (background embedding
job) both depend on a place to store embeddings and a way to compute them. This
plan delivers that foundation so 71-02 can build the recall endpoint and job
without touching schema or provider concerns.

Output: `message_embeddings` table DDL, `embeddings/provider.ts`,
`embeddings/store.ts`, and their RED→GREEN test suites.
</objective>

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Memroos → Ollama | Local HTTP call to the embedding model; Ollama is trusted local infra but may be down. |
| message text → embedding store | Message content is embedded; the BLOB is derived data with no new PII exposure beyond `messages`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-01 | Denial of Service | `embedText` Ollama fetch | mitigate | 5s `AbortSignal.timeout` on every call; failure returns a degraded result, never blocks or throws (D-05). |
| T-71-02 | Tampering | `message_embeddings` DDL | mitigate | DDL is additive `CREATE TABLE IF NOT EXISTS` only — no destructive migration; FK `ON DELETE CASCADE` keeps embeddings consistent with `messages`. |
| T-71-03 | Information disclosure | embedding BLOB | accept | Embeddings are derived from `messages` content already stored in `conversations.db`; no new disclosure surface, no network egress beyond local Ollama. |
| T-71-SC | Tampering | npm installs | mitigate | No new npm packages — `better-sqlite3` and `fetch` are already in the stack; no package-legitimacy checkpoint needed. |

</threat_model>
