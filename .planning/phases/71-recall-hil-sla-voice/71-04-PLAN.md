## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| escalations API → dashboard | Server-rendered escalation data crosses into the client component; `sla_deadline` is a server-trusted timestamp. |
| client clock → countdown | The countdown uses the browser clock; clock skew is cosmetic only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-12 | Information disclosure | escalations page | accept | The page already requires a `reviewer`-or-higher session (existing `/api/escalations` `requireRole`); this plan adds no new data fields, only client-side rendering of already-authorized data. |
| T-71-13 | Tampering | client-side countdown | accept | The countdown is presentation only; authoritative SLA breach state comes from the server (`status`, the 71-03 scheduler). A manipulated client clock changes only the displayed color, never the persisted escalation state. |
| T-71-14 | Denial of Service | per-second setInterval | mitigate | The `setInterval` is cleared on unmount in the `useEffect` cleanup; only mounted cards tick — no leaked timers. |
| T-71-SC | Tampering | npm installs | mitigate | No new npm packages; no package-legitimacy checkpoint needed. |

</threat_model>
