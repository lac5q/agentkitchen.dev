<objective>
Add per-interrupt-type SLA escalation actions and a 60-second background
scheduler that proactively acts on expired HIL tasks.

Purpose: Phase 64 already gave each escalation type a configurable SLA duration
(`getSlaSeconds`) and an ISO `sla_deadline` column. HIL-04 additionally requires
each interrupt type to carry a configurable escalation action; HIL-05 requires a
background scheduler (not just a lazy GET-time check) that polls every 60s and
triggers `notify`, `auto-resolve`, or `abandon`. This plan delivers the action
config, the action engine, and the scheduler.

Output: `getSlaAction()` in `sla-config.ts`, `hil/sla-actions.ts` (action
engine), `hil/sla-scheduler.ts` (60s loop), and `instrumentation.ts`
registration.
</objective>

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduler → hil_escalations | The background poll mutates escalation status and writes audit entries on behalf of `system`. |
| memroos.eval.yaml → action config | The SLA action config is operator-controlled config, not untrusted input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-08 | Elevation of privilege | `auto-resolve` action | mitigate | Auto-resolve uses a fixed `actorId: "system"` and writes a `hil.resolved` audit entry — the resolution is attributable and recorded in the append-only audit log; it cannot be triggered by an unauthenticated caller (scheduler-only path). |
| T-71-09 | Repudiation | SLA action without trail | mitigate | Every SLA action (`notify`/`auto-resolve`/`abandon`) writes a distinct audit entry (`hil.sla_notified`/`hil.resolved`/`hil.sla_abandoned`) so the action is non-repudiable. |
| T-71-10 | Tampering | duplicate action on the same escalation | mitigate | Action runs only on rows still in `open` status; the status transition inside the same transaction makes each escalation act-once — re-polling cannot double-act. |
| T-71-11 | Denial of Service | scheduler exception | mitigate | `runSlaPoll` wraps `runSlaActions` in try/catch; per-escalation actions are individually wrapped so one failure does not abort the batch or kill the timer. |
| T-71-SC | Tampering | npm installs | mitigate | No new npm packages; no package-legitimacy checkpoint needed. |

</threat_model>
