# Phase 83 Plan 01: Memory Inventory + Listing Clarity

## Product Goal

Make the Memory surface explain exactly what it is counting and listing across vector memory, ingested messages, consolidated insights, episodic writes, graph facts, and knowledge files.

## Scope

1. Add a canonical memory inventory vocabulary shared by API responses, UI copy, and tests.
2. Add `/api/memory-inventory` with source-backed category counts and honest degraded states for backend counts that are unavailable.
3. Add provenance listing rows for local/source-owned stores: SQLite messages, consolidated insights, agent memory writes, and knowledge files.
4. Update the Memory page to show category-specific counts, category filters, backend/source/provenance row details, and degraded-state explanations.
5. Add seeded API and component tests that prevent ambiguous "Total Memories" copy and hardcoded demo counts.

## Verification Plan

- RED: add tests for `/api/memory-inventory` category counts, provenance rows, degraded vector-count handling, and Memory list rendering.
- GREEN: implement the route, shared inventory builder, React hook, Memory page summary cards, filter controls, and provenance list rendering.
- Gate: run targeted Vitest, `npm --prefix apps/memroos run typecheck`, lint, full build, GitNexus detect changes, and local/public deploy probes.

## Operator Representation

- Memory page count cards use explicit category labels and source-of-truth tooltips.
- Inventory rows show category, backend, source, project/workspace, timestamp, security label snapshot, consolidation state, salience/access metadata, and evidence pointer when available.
- Backend count gaps render as degraded warnings rather than fabricated totals.
