# Phase 70 Plan: Graph-Native Memory Foundation + Engine Core

## Product Goal

Make MemroOS's memory layer modular and graph-native without replacing the existing production stack. Phase 70 should ship the planned HIL/retry/rollback foundation and upgrade the memory substrate so MemroOS can use the best ideas from `neo4j-labs/agent-memory`: connected short-term, long-term, and reasoning memory; idempotent graph adoption; reasoning-to-entity audit edges; buffered graph writes; and memory evals that test retrieval, audit completeness, and preference/user scoping.

## Requirements Covered

- HIL-01: Operator can modify declared task state fields before resuming a paused LangGraph thread
- HIL-02: Edited values are validated against `OrchestrationState`
- HIL-03: Audit log records HIL edits with actor and before/after values
- ORCH-08: Each hop in a multi-agent chain has a retry budget
- ORCH-09: Forward actions declare compensating actions in lineage rows
- ORCH-10: A2A task status reports granular failed-hop and compensated-hop state
- MEM-06: `MemoryAdapter` exposes only `search()`, `write()`, and `health()`
- MEM-07: Adapter registry maps `MemoryTier` to registered adapters
- MEM-08: Existing mem0/Qdrant/Neo4j backends are concrete adapters

## Repo Findings

MemroOS already has the right broad shape, but the graph tier is too thin:

- `apps/memroos/src/lib/memory/backends.ts` directly queries Neo4j over HTTP and directly calls mem0 for vector search.
- `apps/memroos/src/app/api/memory/add/route.ts` authenticates and forwards writes to mem0 with tier metadata, but graph semantics are not first-class.
- `apps/memroos/src/app/api/memory/multi-search/route.ts` normalizes vector, graph, and episodic results, but it does not assemble connected graph context.
- `apps/memroos/src/lib/memory-consolidation.ts` writes LLM summaries into SQLite, not graph entities/relationships.
- `apps/memroos/src/lib/memory-recall-evals.ts` has useful recall metrics, but it does not yet test graph adoption, reasoning audit paths, or preference fidelity.
- `services/memory/mem0-server.py` has a queue-preserving write path, but buffered graph writes and background write errors are not exposed as adapter capabilities.

The right upgrade is not "swap in Neo4j Agent Memory everywhere." The repo should use its patterns behind a MemroOS adapter and prove value in shadow mode.

## Tasks

1. Add RED tests for the memory adapter contract.
   - `MemoryAdapter` exposes only `search(input)`, `write(input)`, and `health()`.
   - No adapter method returns a Qdrant, Neo4j, mem0, or sqlite client handle.
   - Registry lookup supports multiple adapters per tier and filters by capability.
   - Existing `/api/memory/search`, `/api/memory/graph`, `/api/memory/multi-search`, and `/api/memory/health` response shapes remain compatible.

2. Implement the adapter interface and registry.
   - Add `apps/memroos/src/lib/memory/adapter.ts` with `MemoryAdapter`, `MemorySearchInput`, `MemoryWriteInput`, `MemoryHealth`, `MemoryCapability`, and normalized result types.
   - Add `apps/memroos/src/lib/memory/registry.ts` with tier/capability lookup and default registry construction.
   - Refactor `apps/memroos/src/lib/memory/backends.ts` into concrete adapters for `mem0-qdrant`, `neo4j-direct`, and `sqlite-episodic`.
   - Update memory API routes to resolve through the registry instead of importing concrete backend helpers directly.

3. Add a context-graph shadow adapter.
   - Implement `context-graph` as an optional graph adapter selected by `MEMROOS_CONTEXT_GRAPH_ENABLED=1`.
   - Start with a Python bridge service if using `neo4j-agent-memory`; otherwise implement the same Cypher concepts directly in TypeScript.
   - Pin the library dependency to an exact version or commit during the experiment, because upstream is active and the public release surface is still moving.
   - The adapter must support shadow writes and shadow search without affecting default search results until evals pass.

4. Add graph adoption and schema guardrails.
   - Add a dry-run adoption script or internal endpoint that reports existing Neo4j labels, candidate name properties, already-adopted nodes, skipped nodes, and would-migrate counts.
   - Adopt the `:Entity` super-label/id/type/name pattern only through an idempotent migration.
   - Store adoption reports in SQLite audit or a small local JSON artifact so operators can review the migration before enabling writes.
   - Never mutate labels from user-provided strings without an allowlist or safe identifier validation.

5. Add reasoning memory and `:TOUCHED` audit paths.
   - Extend orchestration/tool execution logging so successful reasoning steps and tool calls can write graph trace records.
   - Add direct `(:ReasoningStep)-[:TOUCHED]->(:Entity)` edges for entities referenced by tool arguments or results.
   - Surface failures as non-blocking memory write errors; memory graph writes must not break agent execution loops.
   - Add a minimal audit query endpoint/helper that can answer: "which reasoning steps touched this entity?"

6. Add buffered write semantics as an adapter capability.
   - Support `writeMode: "sync" | "buffered"` for graph-capable adapters.
   - Buffered mode must expose queue depth, backpressure threshold, flush behavior, and captured background errors.
   - Integrate these fields into `/api/memory/health` so the NOC can show degraded memory persistence separately from backend reachability.
   - Keep default mode sync until shadow-mode evals pass.

7. Expand memory evals.
   - Keep existing recall@k, precision@k, MRR, latency, and tier coverage metrics.
   - Add graph retrieval cases with expected entity IDs.
   - Add audit completeness cases that verify expected `:TOUCHED` paths exist.
   - Add preference/user-scope cases if the adapter enables tenant-scoped preferences.
   - Store eval details in existing `memory_eval_*` tables and bridge aggregate results into the Phase 57 eval engine as already started by `memoryEvalRunToAgentTrace()`.

8. Implement HIL edit-and-continue and orchestration rollback foundation.
   - Add WAL + `busy_timeout=5000` to `OrchestrationStore.__init__` before any schema writes.
   - Pin `langgraph>=1.2,<2.0` if `RetryPolicy` is used.
   - Add validated HIL state edit endpoint in Python and a guarded Next.js proxy route.
   - Use `graph.update_state(..., as_node="route_policy")` so the graph re-enters routing instead of terminating at `approval -> END`.
   - Add retry budget and compensation lineage rows for multi-hop rollback. Compensation must be declarative data in `orchestration_lineage`, not Python closures.

9. Update UI and operator visibility.
   - Extend the Memory Intelligence/NOC panels with adapter health, context-graph shadow status, graph adoption status, buffered queue depth, and latest graph-memory eval score.
   - Add HIL edit controls to the orchestration panel with validation errors and before/after audit evidence.
   - Show failed-hop and compensated-hop details in the dispatch/lineage drawer.

10. Documentation and runbooks.
   - Update memory architecture docs to state the stable contract: MemroOS API and adapter registry are the product surface; backend libraries are implementation details.
   - Document how to run graph adoption dry-run, enable shadow graph memory, run memory evals, flush buffered writes, and rollback to the default adapters.

## Risk Notes

- `neo4j-agent-memory` is useful but moving quickly. The public GitHub repo shows active work and no GitHub releases; treat it as a reference architecture and optional adapter, not a mandatory runtime dependency until pinned and tested.
- Direct Neo4j adoption can create duplicate or malformed graph nodes if labels/properties are guessed. Mitigation: dry-run first, safe identifier validation, explicit label map, and idempotent adoption.
- Graph reasoning writes can slow agent loops. Mitigation: buffered writes are optional, have backpressure, expose errors, and default to sync until evaluated.
- Refactoring memory routes through a registry can accidentally change API response shapes. Mitigation: contract tests around existing route outputs before refactor.
- HIL edit/resume can terminate early if `as_node="approval"` is used. Mitigation: tests must assert edited tasks resume through routing and dispatch.
