## Problem (verified)

SEAL's "apply a proposal then rerun evals, keep if W did not regress" loop
cannot demonstrate any real composite-W movement in production.

`apps/kitchen/src/lib/seal/service.ts` → `applyProposal` (≈line 196):
1. `entry.applyShadow(proposal.diff)` — applies the mutation
2. `postRun = await this.evalService.runForTrace(proposal.traceId, proposal.agentId)`
3. `kept = postRun.compositeW >= proposal.baselineW`

But `EvalService.runForTrace` in `apps/kitchen/src/lib/evals/service.ts`
(≈line 111) **clones the latest baseline run** with a new id/timestamps — it
does NOT re-score anything:

```ts
runForTrace(traceId, agentId?) {
  const baseline = this.getLatestRunForTrace(traceId);
  if (!baseline) throw new Error(`No eval run found for trace ${traceId}`);
  if (agentId && baseline.agentId !== agentId) throw new Error(...);
  return { ...baseline, id: `seal-rerun-${crypto.randomUUID()}`,
           startedAt: now, completedAt: now };
}
```

So `postRun.compositeW === baseline.compositeW` always → W delta is always 0.
The keep/rollback machinery works; the signal it acts on is dead. The W-lift
evidence in `src/lib/seal/__tests__/seal-substrate.test.ts` only exists because
that test injects a **mock** `evalService` returning a hand-set `postRun`.

## Scope of THIS plan (Tier 1 only)

Make `runForTrace` (or a SEAL-specific re-scoring path) produce a **real,
deterministic** `EvalRunResult` whose `compositeW` is a function of
`(baseline, applied proposal diff/forecast)`, using the already-proven pattern
in `apps/kitchen/src/lib/memory-policy-lab.ts`.

**Out of scope (Tier 2 → v3, see `.planning/notes/` v3 work):**
`agent_instruction_patch` / `skill_addition` producing real W lift. Those only
change behavior if the agent re-executes the task; the eval harness scores a
fixed recorded trajectory. Modeling that (synthetic behavior model or agent
re-execution) is the v3 self-improvement loop. This plan must NOT attempt it —
just ensure those proposal types degrade gracefully (W unchanged, kept=true,
clearly logged as "not modeled").

## Why a synthetic deterministic model (honesty constraint)

The proposal `diff` objects are placeholder shells. Example from
`apps/kitchen/src/lib/seal/proposal-registry.ts` `salience_update.buildDraft`:

```ts
diff: { kind:"salience_update", memoryId:null, currentSalience:null,
        proposedSalience:null, decayTier:"standard", note:"..." }
forecastWDelta: 0.02
```

The diffs carry no concrete values to "really" recompute from. So Tier 1 is an
honest **modeled** delta, not an empirical re-run: derive a deterministic W
from baseline + proposal type + `forecastWDelta` + a stable hash of the diff,
exactly like the policy lab derives per-variant W from a config hash. Document
it as modeled in code comments and in `58-01-SUMMARY.md`. Do not overclaim.

## The proven pattern to mirror

`apps/kitchen/src/lib/memory-policy-lab.ts` lines ~76–130:
`buildSyntheticTrace(variant, count)` builds an `AgentEvalTrace` whose
`memory.recallAtK/precisionAtK/mrr` and `outcome` are deterministic functions
of `hashVariantConfig(variant.config)` via `pseudoScoreFromHash(hash,min,max)`,
then feeds it to `scoreTraceWithEvalEngine({trace, config, goldenSet})`
(`apps/kitchen/src/lib/evals/engine.ts` line 82). That function is deterministic
and service-free (judge uses a deterministic hash fallback). Reuse this exact
approach.

## Implementation steps

1. **Add a re-scoring helper** (new file
   `apps/kitchen/src/lib/seal/rescore.ts`):
   `rescorePostApply(opts: { baseline: EvalRunResult; proposalType: string;
   diff: unknown; forecastWDelta: number; config: EvalConfig; goldenSet:
   GoldenSetExample[] }): EvalRunResult`.
   - Build a synthetic post-apply `AgentEvalTrace` seeded from
     `baseline.traceId` + a stable hash of `JSON.stringify(diff)` +
     `forecastWDelta`, mirroring `buildSyntheticTrace`/`pseudoScoreFromHash`
     (copy the deterministic hash + pseudo-score helpers; do not import private
     fns — replicate them or export them from policy-lab).
   - For memory/config proposal types
     (`memory_rewrite`,`query_hint`,`salience_update`,`tier_route`,
     `eval_case_addition`, plus any L3/weight types): nudge the synthetic
     trace's scoring inputs so `scoreTraceWithEvalEngine` yields
     `compositeW ≈ clamp(baseline.compositeW + signedDelta, 0, 1)` where
     `signedDelta` is derived deterministically from the diff hash in
     `[-forecastWDelta, +forecastWDelta*1.5]` (allow regressions so rollback is
     still exercised).
   - For `agent_instruction_patch`/`skill_addition`/`noop_test`: return a clone
     of `baseline` (W unchanged) but stamp `scorerResults`/judge detail with
     `{ wLiftModeled: false, reason: "behavioral effect not modeled (v3)" }`.
   - Call `scoreTraceWithEvalEngine({ trace: syntheticTrace, config,
     goldenSet })` and return its `EvalRunResult` (real layers/judge/driftGuard,
     just driven by the synthetic trace). `EvalRunResult` shape is in
     `apps/kitchen/src/lib/evals/types.ts` line 195.

2. **Wire it into the SEAL eval path.** In
   `apps/kitchen/src/lib/evals/service.ts`, change `EvalService.runForTrace`
   so that when a re-score context is available it returns
   `rescorePostApply(...)` instead of the baseline clone. Cleanest seam:
   `applyProposal` already has `proposal.proposalType`, `proposal.diff`,
   `proposal.forecastWDelta`, `proposal.baselineW`, `proposal.baselineLayers`.
   Either (a) add an optional 4th arg to the `EvalServiceLike.runForTrace`
   signature (`apps/kitchen/src/lib/seal/service.ts` line ~47) carrying the
   proposal context and have `applyProposal` pass it, or (b) add a dedicated
   `EvalServiceLike.rescoreForProposal(proposal, baseline)` method and call it
   from `applyProposal` instead of `runForTrace`. (b) is cleaner — do (b).
   Update the interface, `EvalService`, and the SDK impl
   (`apps/kitchen/src/lib/seal/sdk-eval-service.ts`). The
   `createEvalServiceForSeal()` fallback path (`apps/kitchen/src/lib/seal/
   apply.ts`) must keep working with no server.

3. **Load the golden set + config** for re-scoring from the same source the
   engine uses: `loadEvalConfig()` (`apps/kitchen/src/lib/evals/config.ts`) and
   `loadGoldenSet(goldenSetPathForTrace(config, {agentId, role}))`
   (`apps/kitchen/src/lib/evals/golden-sets.ts`). Golden sets are populated
   (see `golden-sets/*.jsonl`, generator `golden-sets/.generate.mjs`).

      eval service (real `SealService` + real re-score path).
- [ ] Instruction/skill proposals return unchanged W, flagged
      `wLiftModeled:false` — no overclaiming.
- [ ] Determinism: same (baseline,diff) → same W across runs.
