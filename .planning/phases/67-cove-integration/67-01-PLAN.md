# Phase 67 Plan 01: CoVe Integration

## Product Goal

CoVe ships as a callable four-step agent runtime pipeline and as a registered
eval scorer that measures hallucination reduction against a baseline trace,
while remaining provider-neutral across Claude, Ollama/Hermes, and
OpenAI-compatible endpoints.

## Requirements Covered

- COVE-01: callable CoVe wrapper
- COVE-02: registered eval scorer
- COVE-03: provider-neutral endpoint support

## Implementation Plan

1. Add RED tests for the CoVe pipeline.
   - Draft answer generation.
   - Verification-question generation.
   - Independent fact-check answers.
   - Revised answer with trace metadata.

2. Implement `apps/kitchen/src/lib/cove/`.
   - Define config, provider, trace, and result types.
   - Implement `cove(agentFn, config)` and a lower-level `runCovePipeline`.
   - Support sequential and batched verification calls.

3. Add provider-neutral LLM caller.
   - Support injected test clients.
   - Support OpenAI-compatible endpoint shape for Ollama/vLLM/Hermes.
   - Keep Claude/provider-specific logic out of the core pipeline by adapting
     to the common call interface.

4. Register eval scorer.
   - Add a `cove_hallucination_delta` scorer to the eval scorer registry.
   - Compare baseline trace and CoVe trace for unsupported claims, corrections,
     and confidence metadata.
   - Return score and metadata suitable for W dashboards.

5. Extend configuration.
   - Add `cove.enabled`, `cove.max_verification_questions`,
     `cove.parallel_verification`, and `cove.judge_endpoint`.
   - Validate bounds and preserve disabled-by-default behavior.

6. Add API/demo route.
   - Add a small local demo route or test harness that runs a factual prompt
     with and without CoVe and returns both traces.
   - Keep external LLM calls disabled unless explicitly configured.

7. Update docs and planning summary.
   - Document provider configuration, expected trace shape, and scorer output.
   - Record verification evidence and any deferred live-provider checks.
