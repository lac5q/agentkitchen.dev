# Phase 77 Plan 01 — Safe Index Projections + Envelope Encryption

## Goal

Close the first concrete slice of MEMSEC-05, MEMSEC-06, and MEMSEC-07 by making derived embeddings provenance-bearing and protecting restricted vault artifacts with app-level envelope encryption.

## Requirements

- MEMSEC-05
- MEMSEC-06
- MEMSEC-07

## Implementation Plan

1. Add vault envelope encryption.
   - Local file-backed key provider at `MEMROOS_VAULT_KEY_PATH` or `~/.memroos/vault.key`.
   - AES-256-GCM encrypted payloads and wrapped data keys.
   - Rewrap helper for key-encryption-key rotation without rewriting plaintext.

2. Wire encryption into raw vault writes and reads.
   - Restricted labels (`private`, `sealed`, review-required, redaction-required, or sensitive) are encrypted before compression.
   - Public approved/indexable artifacts remain zstd-only for performance and indexability.
   - Existing zstd-only artifacts remain replay-compatible.

3. Add embedding provenance fields.
   - `artifact_id`, `source_span`, `modality`, `model_version`, and `label_version` columns on `message_embeddings`.
   - `upsertEmbedding()` stores provenance while preserving the old call signature.

4. Verify with focused tests, typecheck, lint, and build.

## Remaining Phase 77 Work

- Qdrant/Neo4j/qmd eviction or redacted-projection sweep for already-indexed restricted content.
- Operator-facing key backup/restore drill.
- Full index rebuild path after reclassification.
