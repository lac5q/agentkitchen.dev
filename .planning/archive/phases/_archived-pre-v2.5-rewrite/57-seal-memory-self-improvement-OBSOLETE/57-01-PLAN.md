<objective>
Implement a SEAL-inspired memory recollection improvement loop for MemroOS.

In this plan, SEAL means Self-Adapting Language Models, but v1 does not fine-tune weights. Instead, failed recall evals and operator-marked bad recalls create reflections and typed self-edit proposals. Operators approve or reject each proposal. Approved proposals are applied to memory configuration, memory records, eval cases, routing hints, salience metadata, or agent instructions only through explicit gates. After application, the system reruns memory evals and keeps the change only if recall metrics improve or remain above threshold.
</objective>

Memory improvement proposals need a typed JSON payload and a database-backed audit trail.

Karpathy/autoresearch inspiration: keep the loop tiny and honest. In autoresearch, the fixed harness and eval infrastructure stay stable while the agent edits one mutable file, runs a fixed-time experiment, scores one objective, and keeps only improvements. In MemroOS v1, the mutable target is a memory policy candidate, not model weights. The objective is a scalar memory score derived from recall eval metrics.

<requirements>

**SEAL-01 — Reflection From Recall Failure**

Given a failed `MemoryEvalRun` or an operator-marked bad recall, the system creates a reflection record containing:

- trigger type: `eval_failure` or `bad_recall`
- source run id and case id when available
- failed metrics, tier failures, retrieved top-K summary, and missing expected facts
- deterministic root-cause category:
  - `tier_unavailable`
  - `no_relevant_result`
  - `late_recall`
  - `low_precision`
  - `wrong_tier`
  - `stale_or_low_salience`
  - `missing_eval_coverage`
  - `agent_instruction_gap`
- optional provider-authored explanation when an LLM is available

The deterministic path is required so the loop still works during Gemini quota exhaustion.

**SEAL-02 — Typed Self-Edit Proposals**

Each reflection can generate one or more typed proposals:

- `memory_rewrite`: rewrite or compact a memory record while preserving provenance
- `query_hint`: add retrieval-time query expansion/hint metadata for an agent, topic, or case
- `salience_update`: raise/lower salience tier or access-resistance metadata for a memory
- `tier_route`: adjust future write/read routing between vector, graph, episodic, and qmd
- `eval_case_addition`: add a new regression case to `evals/memory-recall/cases.json`
- `agent_instruction_patch`: propose a patch to an agent or skill instruction file

Every proposal must include:

- id, proposal type, status, created/updated timestamps
- source reflection id
- summary and rationale
- risk level: `low`, `medium`, `high`
- target descriptor, not raw arbitrary paths
- typed `patch` JSON
- baseline metrics and acceptance thresholds
- rollback plan
- privacy classification and redaction summary

**SEAL-03 — Human Approval Gate**

No proposal applies automatically. Operator-authenticated actions:

- generate proposals from latest failed eval run
- approve a pending proposal
- reject a pending proposal with optional reason
- apply an approved proposal
- view status history

High-risk proposal types must remain extra conservative:

- `agent_instruction_patch` may write only to allowlisted instruction targets and should default to queued manual review if the target is outside the allowlist.
- `memory_rewrite` must preserve the original content in rollback metadata.
- `tier_route` changes must be scoped by agent/topic/case, not global defaults, unless explicitly marked high risk.

**SEAL-04 — Eval-Gated Keep/Rollback**

After applying a proposal:

- rerun `runMemoryRecallEvalSuite` in the smallest relevant mode:
  - source canary case -> `canary`
  - source gold case -> `gold`
  - scenario or new eval coverage -> `full`
- compare post-apply metrics to baseline
- keep only when:
  - source case passes, and
  - aggregate pass rate does not decline, and
  - p95 latency stays under threshold, and
  - required tier coverage is not reduced
- otherwise rollback the applied mutation and mark the proposal `rolled_back`

The first implementation may make rollback exact for database-backed changes and queue manual rollback instructions for filesystem instruction patches.

**SEAL-05 — Visible, Auditable Loop**

The Library/Ledger memory UI surfaces:

- latest eval status
- pending memory improvement proposals
- approved proposals awaiting application
- kept, rejected, and rolled-back history
- before/after metric deltas
- tier failures and root-cause labels

Docs explain the loop, safety boundaries, and why it is SEAL-style without model weight updates.

**KAR-01 — Scalar Memory Score**

Add `computeMemoryEvalScore(run)` so memory experiments have one objective comparable to autoresearch's `val_bpb`.

The score should be deterministic and built from existing metrics:

- reward: `recallAtK`, `mrr`, `precisionAtK`, `tierCoverage`
- penalty: `falsePositiveRate`, latency over threshold, aggregate pass-rate regression
- hard fail: any missing required tier coverage for the source case

The first implementation may use:

```text
score =
  2.0 * recallAtK
  + 1.0 * mrr
  + 0.75 * precisionAtK
  + 0.5 * tierCoverage
  - 0.25 * falsePositiveRate
  - latencyPenalty
```

**KAR-02 — Fixed Memory Policy Lab Harness**

Add a lab harness that prepares fixture eval data, records a baseline, runs the candidate policy, reruns the same eval mode, and records accept/reject. The harness owns setup, scoring, timeout, event logging, and rollback. Candidate policy code must not edit eval data, scoring, or history.

**KAR-03 — One Mutable Policy File**

Create one narrow candidate module for experiments. The first candidate hooks should be policy-shaped, not freeform:

- query expansion hints
- tier read order
- result reranking weights
- salience boost policy
- fallback route when vector/graph tier is unavailable

The lab must reject experiments that touch files outside the allowlisted candidate module unless the operator explicitly moves the proposal into manual review.

**KAR-04 — Experiment Ledger and Audit Surface**

Persist every experiment with:

- experiment id
- actor
- baseline run id and score
- candidate run id and score
- changed file hash
- eval mode
- status: accepted, rejected, errored, rolled_back
- top failure cases and root causes
- created/completed timestamps

The Recall Quality UI should show experiment history separately from human proposals so the operator can tell policy experiments from applied memory edits.

**KAR-05 — Autoregression Deferred Boundary**

Do not train AR model weights in v1. Store clean eval/proposal/experiment trajectories so a later phase can train a lightweight AR memory predictor on audited data. Any future AR predictor must be advisory until it beats the recall eval baseline and passes the same human/eval gates.

<risk_controls>

- No model weight fine-tuning in v1.
- No AR model training in v1; collect clean experiment trajectories first.
- No automatic instruction patching outside explicit allowlists.
- No raw secrets, full tool outputs, or proposal bodies stored as memory.
- No global tier-routing changes without high-risk labeling.
- No direct Qdrant writes; vector writes continue through mem0 HTTP APIs.
- Failed post-apply evals roll back or queue manual rollback.
- Gemini quota exhaustion is treated as a captured reliability signal, not an implementation blocker.
- Karpathy-style experiments can mutate only the dedicated memory policy candidate file.

</risk_controls>
