# Phase 57 Research: Autoresearch-Style Memory Learning

## Research Question

How should MemroOS add Karpathy-style self-learning autoregression to an agentic memory system without losing auditability, safety, or GSD execution discipline?

## Primary Source

Karpathy's `autoresearch` repo describes a deliberately small autonomous research loop:

- `prepare.py` is fixed data prep and evaluation infrastructure.
- `train.py` is the single mutable file the agent edits.
- `program.md` is the human-authored instruction file.
- Each experiment runs for a fixed five-minute wall-clock training budget.
- The objective metric is `val_bpb`, validation bits per byte; lower is better.
- The loop keeps improvements and discards regressions.

Source: https://github.com/karpathy/autoresearch

## Translation to MemroOS

The equivalent MemroOS loop should not start by training model weights. The best v1 target is memory-management policy because the repo already has:

- A three-tier memory stack: vector, graph, episodic.
- A first-party recall eval harness with `recallAtK`, `precisionAtK`, `mrr`, `tierCoverage`, false-positive rate, and latency.
- API and CLI eval surfaces.
- Existing human-gated proposal UX patterns from APO.

The clean mapping is:

| Autoresearch | MemroOS Memory Loop |
| --- | --- |
| `program.md` | Memory lab instruction file and GSD phase plan |
| `train.py` | One mutable memory policy module |
| `prepare.py` | Fixed eval cases, fixtures, scoring, and rollback harness |
| `val_bpb` | Scalar memory score derived from recall eval metrics |
| 5-minute training run | 2-5 minute recall eval experiment |
| Keep/discard code diff | Keep/rollback approved memory proposal |

## Option Matrix

### Option 1: Metric-Gated Memory Proposal Loop

Failed recall evals create reflections and typed self-edit proposals. Human approval gates application. Post-apply evals decide keep or rollback.

Recommendation: **Implement first.** This is already aligned with Phase 57 and uses the existing eval harness.

Why it wins now:

- No GPU required.
- Works during Gemini/vector quota failures.
- Directly improves production memory behavior.
- Easy to audit through SQLite proposal/event tables.

### Option 2: Single-File Memory Autoresearch Sandbox

Create a tiny sandbox where an agent may edit only one file, such as `apps/memroos/src/lib/memory-policy-lab.ts`, then run fixed recall evals against fixture data. The harness accepts or rejects the edit by scalar score.

Recommendation: **Implement after Option 1 as the Karpathy-style GSD feature.**

Why:

- Preserves the most important autoresearch constraint: one mutable file.
- Makes experiments reviewable and reversible.
- Lets GSD run overnight-style loops without touching production routing directly.

### Option 3: Lightweight AR Memory Predictor

Train a small autoregressive predictor on normalized memory trajectories, such as `state -> action -> outcome -> reflection`. Use it to predict likely needed memories or future recall failures.

Recommendation: **Defer until the eval/proposal loop is stable.**

Why:

- It needs GPU or ML runtime decisions that MemroOS does not currently require.
- It can help proactive retrieval, but it is not the shortest path to reliable recall.
- It should consume audited trajectory data generated by Options 1 and 2.

### Option 4: Trajectory Self-Training / STaR-Style Loop

Generate candidate improved trajectories, filter by downstream task success, and train the predictor on accepted trajectories.

Recommendation: **Defer.**

Why:

- Needs a reliable downstream task-success evaluator.
- Higher risk of synthetic-data feedback loops.
- Useful only after recall evals and proposal history are strong.

### Option 5: Full Memory Research Org

Run multiple agents: one proposes memory policy changes, one executes experiments, one audits failures, one writes docs/proposals.

Recommendation: **Defer until Phase 57 v1 has shipped.**

Why:

- More powerful, but also more coordination overhead.
- The GSD system already provides phase decomposition; first make the single-loop path dependable.

## Recommended Architecture

Phase 57 should ship in two layers.

Layer A: Human-Gated Memory Self-Improvement

- Failed eval or bad recall creates reflection.
- Reflection creates typed proposal.
- Operator approves or rejects.
- Approved proposal applies to memory policy/config/eval data.
- Recall eval reruns.
- Change is kept only if metrics pass.

Layer B: Autoresearch-Inspired Memory Lab

- A fixed harness prepares fixture data and baseline metrics.
- One mutable policy file is the only experiment target.
- A scalar score decides accept/reject:

```text
memory_score =
  2.0 * recallAtK
  + 1.0 * mrr
  + 0.75 * precisionAtK
  + 0.5 * tierCoverage
  - 0.25 * falsePositiveRate
  - latencyPenalty
```

- The lab writes experiment rows to SQLite:
  - experiment id
  - baseline score
  - candidate score
  - changed file hash
  - eval mode
  - status: accepted, rejected, errored
  - failure summary

## GSD Feature Shape

Add requirements to Phase 57:

- `KAR-01`: memory score aggregator for recall eval metrics.
- `KAR-02`: fixed memory lab harness that runs evals against candidate policy.
- `KAR-03`: one mutable policy module with typed policy hooks.
- `KAR-04`: experiment ledger and audit UI.
- `KAR-05`: hard safety boundaries: no production mutation, no secrets, no model weights in v1.

## Implementation Order

1. Finish Phase 57 SEAL proposal loop.
2. Add `computeMemoryEvalScore(run)` to the self-improvement module.
3. Add a memory lab module with one mutable policy surface.
4. Add a CLI/API action to run a candidate experiment.
5. Store experiment history and surface it in the Recall Quality UI.
6. Only then evaluate a lightweight AR predictor trained on accepted proposal/eval trajectories.

## Recall Eval Audit Loop

Every experiment should produce:

- baseline metrics
- candidate metrics
- kept/rejected decision
- top failing cases
- root-cause labels
- proposed eval case additions
- privacy redaction status

The failure log should feed future learning through `eval_case_addition` proposals, not by silently expanding the eval suite. This prevents the system from grading itself with unreviewed tests.

## Decision

Use Option 1 + Option 2 for v1.

Do not introduce AR model training in the first implementation. Instead, build the eval-gated policy loop in a way that can later train an AR predictor on clean, audited experiment trajectories.
